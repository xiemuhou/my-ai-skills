# Article to Markdown — 执行规范

## 核心原则

1. **原样保存**: 保留原文内容，不做修改或润色
2. **原图链接**: 图片使用原始 URL，不下载、不转存、不改写
3. **特色图片优先**: 提取文章特色图片链接，在 YAML Front Matter 结束后的正文第一行显示特色图片
4. **YAML Front Matter**: 文件第一行插入元数据头，category 和 tag 根据文章内容自动生成
5. **本地存储**: 保存到用户指定目录，未指定则保存到当前工作目录
6. **标题不入正文**: 文章标题仅用于生成文件名，不出现在 Markdown 正文中
7. **标题层级修正**: 文章无一级标题时，所有标题自动提升一级

---

## 执行流程

### 阶段 1: 获取参数

向用户确认以下信息（用户未指定则由 AI 根据文章内容自动生成）：

| 参数 | 生成方式 | 说明 |
|------|----------|------|
| `category` | AI 根据文章内容推断 | 文章分类，最多 2 个词语 |
| `tag` | AI 根据文章内容提取 | 内容标签，最多 2 个词语 |
| `status` | `publish` | 发布状态 |
| `filename` | 自动从标题提取 | 保存的文件名 |

用户可在提供 URL 时一并指定：
- `保存文章 https://xxx.com/xxx --category 技术 --tag python,AI --filename my-note`
- `保存文章 https://xxx.com/xxx --featured-image https://xxx.com/cover.jpg`

### 阶段 2: 获取网页内容

1. 使用 `WebFetch` 工具获取 URL 对应网页内容；如果页面结构复杂，优先获取原始 HTML 再自行提取正文，不要只依赖可见正文转换结果。
2. 对 WordPress 站点必须检查是否存在文章 ID，并优先尝试 WordPress REST API 完整正文：
   - 从 `body` class 中识别 `postid-<id>`
   - 或从 `link[rel="shortlink"]` 的 `?p=<id>` 识别
   - 或从 canonical URL 末尾数字识别
   - 请求 `/wp-json/wp/v2/posts/<id>`，优先使用 `content.rendered` 作为正文来源
3. 如果 REST API 正文长度明显大于前端页面正文，必须使用 REST API 内容；这是 WordPress 主题、Argon 折叠块、懒加载和前端截断场景下避免漏正文的关键规则。
4. 同时尽量从页面 HTML、REST API 元数据中识别特色图片，例如 `og:image`、`twitter:image`、`image_src`、JSON-LD `image`、`jetpack_featured_media_url`、`yoast_head_json.og_image`。
5. 如果网页需要认证或获取失败，提示用户并退出

### 阶段 3: 内容处理

对获取到的内容做以下处理：

1. **提取标题**: 取第一个 `#` 或 `<h1>` 标题作为文章标题，用于生成文件名
2. **标题不入正文**: 删除正文中的文章标题行（`# 标题`），不保留在 Markdown 文件中
3. **提取特色图片**: 按优先级选择 `--featured-image`、`og:image`、`twitter:image`、JSON-LD `image`、页面首张大图或文章开头主图作为特色图片链接
4. **保留特色图片 URL**: 保留页面提供的原始图片链接，不下载、不转存、不改写、不补全、不裁剪查询参数
5. **移除来源信息**: 删除 "文章来源"、"原文链接"、"来源"、"转载自" 等行
6. **移除更新时间**: 删除 "最后更新"、"更新时间"、"发布于"、"Posted on"、"本文最后更新于" 等行
7. **移除页脚推广**: 删除文末的 "小提示"、"订阅"、"关注" 等网站推广内容
8. **保留原图链接**: 不做任何修改，保持 `![](url)` 原样
9. **自动生成标签**: 根据文章内容推断 category 和 tag（各不超过 2 个词语）
10. **标题提升**: 如果文章中没有一级标题（`# `），将所有标题提高一级（`## ` → `# `，`### ` → `## `，以此类推）
11. **保留原文内容**: 除上述移除项外，其余内容原样保留

复杂页面保全规则：

- **折叠区块只去外壳，不删正文**：遇到 Argon/Bootstrap/WordPress 折叠块，例如 `.collapse-block`、`.collapse-block-title`、`.collapse-block-body`、`.collapse`、`details` 等，只把折叠标题转为普通文本或加粗提示，并保留折叠块内部以及折叠块后面的全部正文。绝不能用“替换整个折叠容器”的方式处理，否则会误删后续正文。
- **懒加载图片取真实地址**：如果图片 `src` 是 `data:image`、透明占位图或 loading SVG，必须优先取 `data-original`、`data-src`、`data-lazy-src`、父级 fancybox/lightbox 的 `href` 等真实图片链接。
- **不要把占位图写入 Markdown**：最终文章图片不应出现 `![](data:image...)`。如果 `data:image` 出现在代码块中，例如 CSS 背景图示例，则属于原文代码，应保留。
- **完整性校验**：转换后对比标题、正文长度、图片数量、文末标题/相关链接。若转换结果在折叠块附近结束，或明显少于 REST API/原始 HTML 正文，必须重新提取。

特色图片处理规则：

- 如果成功提取特色图片链接，在 YAML Front Matter 结束后的正文第一行写入：`![](<特色图片原始URL>)`
- 特色图片行之后保留一个空行，再写入处理后的正文
- 如果正文开头已经包含同一个特色图片，不要重复插入
- YAML front matter 只允许出现 `category`、`tag`、`status` 三个字段；不允许出现任何图片、封面、头图、缩略图相关字段
- 如果无法可靠识别特色图片，不要编造图片链接，也不要插入占位图片

### 阶段 4: 生成文件名

- 从文章标题提取，去除非法字符（`<>:"/\|?*`）
- 中英文混排时保留原样
- 长度控制在 50 字符以内
- 扩展名 `.md`
- 保存到用户指定目录，未指定则保存在当前工作目录

### 阶段 5: 写入文件

````markdown
---
category: [<用户指定的 category>]
tag: [<用户指定的 tag>]
status: <用户指定的 status>
---

![](<特色图片原始URL>)

<处理后的内容>
````

- 使用 `Write` 工具写入文件
- 写入前必须自检：两个 `---` 之间只能包含 `category`、`tag`、`status`；特色图片必须在第二个 `---` 之后
- 写入前必须做内容完整性自检：
  - WordPress 文章已优先检查 `/wp-json/wp/v2/posts/<id>` 的 `content.rendered`
  - 折叠块标题后面的正文仍然存在
  - 文末内容、总结、相关链接等没有被截断
  - 文章图片没有 `![](data:image...)` 占位图
  - Markdown 文件大小、标题数量、图片数量与原始正文基本匹配
- 写入前检查文件是否已存在，若存在则终止任务并提示"文件已存在: <路径>"

## 参数速查

| 参数 | 效果 |
|------|------|
| `--category <名称>` | 指定文章分类（最多 2 个词语），未指定则由 AI 自动生成 |
| `--tag <标签>` | 指定标签（逗号分隔，最多 2 个），未指定则由 AI 自动生成 |
| `--status <状态>` | 指定发布状态，默认 `publish` |
| `--filename <名称>` | 指定保存文件名，默认从标题提取 |
| `--path <目录>` | 指定保存目录，默认当前工作目录 |
| `--featured-image <URL>` | 手动指定特色图片链接，优先级高于自动提取 |

## 输出格式

```
✅ 文章已保存
📝 标题: <文章标题>
📁 文件: <文件名>.md
📂 路径: <保存目录>/<文件名>.md
🏷️  分类: <category>
📌 标签: <tag>
🖼️  特色图片: <图片链接或 未找到>
```

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| URL 无法访问 | 提示用户检查链接是否有效 |
| 网页需要登录 | 提示用户提供可直接访问的 URL |
| 内容为空 | 提示"未能提取到文章内容"并退出 |
| 文件已存在 | 终止任务，提示"文件已存在: <路径>" |
| 标题无法提取 | 使用 URL 的最后一段作为文件名 |

## 重要约束

1. **不修改原文**: 只移除来源/更新时间/页脚推广/标题，不对正文做任何编辑
2. **不操作图片**: 所有图片保持原始链接，不下载、不转存、不改写、不补全、不压缩、不裁剪参数
3. **不显示来源**: 最终文件中不含文章来源和更新时间
4. **不做 AI 摘要**: 只保存原文，不添加任何 AI 生成的内容
5. **标题不入正文**: 文章标题仅用于 frontmatter 和文件名，不出现在正文中
6. **标签自动生成**: category 和 tag 根据文章内容推断，各不超过 2 个词语
7. **标题层级修正**: 无一级标题时自动提升所有标题层级
8. **编码为 UTF-8**: 中文内容正确保存
9. **特色图片不编造**: 只保存页面真实存在或用户明确指定的图片链接；找不到时跳过
10. **WordPress 完整正文优先**: WordPress 页面必须优先核对 REST API 正文，避免主题折叠、摘要、懒加载或前端截断导致正文缺失
11. **折叠内容保全**: 折叠、隐藏、展开区域的内容属于正文，必须保存；处理时只能展开/去壳，不能删除容器内或容器后的内容
