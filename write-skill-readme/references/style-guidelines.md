## 小白友好设计规范

### 1. 表格化决策指南

**原则**：用表格而非文字解释复杂选择

**示例**（档位选择）：

```markdown
### 如何选择合适的档位？

| 你的需求 | 推荐档位 | 理由 |
|---------|---------|------|
| 投稿 Nature Reviews、Chemical Reviews 等顶刊 | **Premium** | 符合顶刊深度与广度要求 |
| 学位论文的 Related Work 章节 | **Standard** | 多数导师接受此范围 |
| 快速了解某领域研究现状 | **Lite** | 快速获取核心信息 |
```

**示例**（参数对照）：

```markdown
### 三档位对照表

| 档位 | 字数范围 | 参考文献数 | 典型场景 | PDF 页数 | 别名 |
|------|---------|-----------|---------|---------|------|
| **Premium（旗舰级）** | 10000–15000 | 80–150 | Nature Reviews 级别综述 | 16–25 页 | 旗舰级、顶刊级、高级 |
| **Standard（标准级）** | 6000–10000 | 50–90 | 学位论文 Related Work | 10–16 页 | 标准级、常规 |
| **Lite（轻量级）** | 3000–6000 | 30–50 | 快速了解领域现状 | 6–10 页 | 轻量级、入门 |
```

### 2. 丰富别名降低认知负荷

**原则**：为同一概念提供多种称呼方式

**示例**：

```markdown
| 档位 | 别名 |
|------|------|
| Premium | 旗舰级、顶刊级、高级、Full |
| Standard | 标准级、常规、Normal |
| Lite | 轻量级、入门、Basic |
```

**好处**：用户可以用任何熟悉的名称称呼，无需记忆特定术语

### 3. 对话式呈现

**原则**：展示用户与 AI 的实际交互过程

**示例**：

```markdown
### 示例 1：首次发布

```
你：发布 v1.0.0 到 GitHub

技能：检测到这是首次发布，将创建项目第一个 Release。
```
```

**好处**：小白用户可以直观理解"我说什么 → AI 做什么"

### 4. 行内注释的代码块

**原则**：注释在命令前（而非后），阅读时先看注释再看命令

**示例**：

```markdown
```bash
# 默认：同时安装到 Codex 和 Claude Code（仅安装有更新的）
python3 install-bensz-skills/scripts/install.py

# 仅安装到 Claude Code
python3 install-bensz-skills/scripts/install.py --claude

# 仅安装到 Codex
python3 install-bensz-skills/scripts/install.py --codex

# 强制重新安装所有技能（忽略版本检查）
python3 install-bensz-skills/scripts/install.py --force
```
```

**好处**：用户先理解意图，再看具体命令

### 5. 括号补充说明

**原则**：用括号补充额外信息，无需翻阅文档

**示例**：

```markdown
- **仅安装有更新的**（默认）— 跳过已安装的最新版本
- **强制重新安装所有**（忽略版本检查）— 覆盖已安装的版本
```

## Prompt 呈现规范

### 1. 经典 Prompt 优先原则

**原则**：将"最推荐/最简单/作者最喜欢"的 Prompt 放在最前面的"快速开始"章节

**示例**：

```markdown
## 快速开始

- 作者最喜欢用的一版

```
请使用 systematic-literature-review skill 写一篇"HER2-ADC在乳腺癌中的研究进展"的Premium级综述
输入：主题（文字）
输出：该 skill 的默认交付物（见该 skill 的 README/说明）
另外，还有下列参数约束：
- 参考范围：近 2023-2025 年为主（更早但特别重要的也可纳入）
- 结构要求：增加“未来 3 年研究方向”小节
- 输出控制：工作目录名为 HER2-ADC-01
```
```

### 2. 场景化标题

**原则**：为每个 Prompt 标注清晰的使用场景

**示例**：

```markdown
### 示例 1：从自然语言提取（最简单）

### 示例 2：从文件提取

### 示例 3：从图片提取

### 示例 4：从网页 URL 提取
```

**好处**：用户快速找到所需场景

### 3. 渐进式复杂度

**原则**：按"最小可用 → 进阶 → 高级"的顺序排列 Prompt

**示例**：

```markdown
### 示例 1：基础用法（最小可用）

```
简单 Prompt
```

### 示例 2：指定参数

```
带参数的 Prompt
```

### 示例 3：高级用法

```
复杂 Prompt
```
```

### 4. Prompt 编写要点

**要素清单**：

| 要素 | 说明 | 示例 |
|------|------|------|
| **明确目标** | 清楚说明要做什么 | "写一篇 Premium 级综述" |
| **指定范围** | 界定输入/输出范围 | "参考文献以 2023-2025 年为主" |
| **提供上下文** | 给出必要的背景信息 | "基于这个调研报告..." |
| **可选参数** | 用括号标注可选内容 | "(可选)指定工作目录" |

**反模式**：

```markdown
❌ 过于模糊：
"帮我写个综述"

❌ 过于冗长：
"请使用 systematic-literature-review skill 写一篇关于HER2-ADC在乳腺癌中的研究进展的Premium级综述，参考文献以近2023-2025年为主，更早之前的文献如果特别相关特别重要的也可以纳入，要有一个小节专门讨论未来3年较有前景的研究方向，工作目录名为HER2-ADC-01，不要使用过时的文献，确保引用格式正确，输出PDF和Word版本..."
```

## 硬编码用法处理规范

### 1. 明确区分优先级

**原则**：用章节标题明确标注"推荐"vs"备选"

**示例**：

```markdown
## 推荐用法（Prompt 调用 Skill）

## 备选用法（脚本/硬编码流程）
```

### 2. 解释推荐理由

**原则**：告诉用户为什么要优先用 Prompt

**示例**：

```markdown
在 Claude Code / OpenAI Codex CLI 里，通常**优先用自然语言 Prompt** 触发 Skill
（而不是手动跑脚本）。下面这条 Prompt 已验证可靠：
```

### 3. 硬编码用法放在最后

**原则**：将脚本命令放在 README 的最后部分

**章节顺序**：

1. 快速开始（Prompt）
2. 设计理念/功能概述
3. 提示词示例（Prompt）
4. 输出文件/配置选项
5. **备选用法（脚本）** ← 最后
6. FAQ

### 4. 脚本步骤编号

**原则**：用编号列表组织多步骤脚本流程

**示例**：

```markdown
## 备选用法（脚本/硬编码流程）

### 步骤 1：状态检查（看缺什么）

```bash
python3 skills/make_latex_model/scripts/check_state.py projects/NSFC_Young
```

### 步骤 2：分析 Word PDF 基准（提取样式参数）

```bash
python3 skills/make_latex_model/scripts/analyze_pdf.py projects/NSFC_Young/template/word_baseline.pdf
```

### 步骤 3：生成 LaTeX 模板

```bash
python3 skills/make_latex_model/scripts/generate_template.py projects/NSFC_Young
```
```

## 语言风格规范

### 1. 第二人称

**原则**：使用"你"而非"用户"，增强亲和力

**示例**：

```markdown
✅ 好：你：发布 v1.0.0 到 GitHub

❌ 差：用户：发布 v1.0.0 到 GitHub
```

### 2. Emoji 使用规范

**有选择地使用**，不要过度：

| 场景 | 推荐 emoji | 示例 |
|------|-----------|------|
| 特性列表 | ✨、🚀、🎯 | ✨ 核心特性 |
| 状态标记 | ✅、❌、⏭️ | ✅ 已安装 |
| 章节标题 | 📖、💡、🔧 | 📖 快速开始 |
| 警告/注意 | ⚠️、⚡ | ⚠️ 注意 |

**注意**：NSFC 相关技能更保守，减少 emoji 使用

### 3. 技术术语高亮

**原则**：用代码高亮区分文件、路径、命令

**示例**：

```markdown
- 文件：`` `SKILL.md` ``
- 路径：`` `/path/to/file` ``
- 命令：`` `python3 scripts/run.py` ``
- 参数：`` `--force` ``
```

### 4. 避免说教

**原则**：用表格、列表而非长篇大论

**示例**：

```markdown
✅ 好（表格）：
| 档位 | 适用场景 |
|------|---------|
| Premium | 顶刊投稿 |
| Standard | 学位论文 |

❌ 差（长文）：
Premium 档位适用于您计划向 Nature Reviews、Chemical Reviews 等顶级期刊投稿的情况，
因为此类期刊通常要求综述具有极高的深度和广度...（省略 500 字）
```

## 静态检查清单

在完成 README 编写后，使用以下清单进行自检：

### 内容完整性检查

- [ ] 标题与受众声明
- [ ] 快速开始章节（含常规 Prompt）
- [ ] 进阶 Prompt（带“参数约束”，可选）
- [ ] 场景化变异 Prompt（按需补充 0-2 个）
- [ ] 设计理念/功能概述
- [ ] 使用示例（如有）
- [ ] 输出文件说明
- [ ] 常见问题（FAQ）
- [ ] 指向其他文档的链接

### 小白友好检查

- [ ] 是否用表格呈现复杂选择？
- [ ] 是否提供丰富别名？
- [ ] 是否用对话式呈现交互？
- [ ] 代码块是否有行内注释？
- [ ] 是否用括号补充说明？

### Prompt 质量检查

- [ ] 经典 Prompt 是否在最前面？
- [ ] 每个 Prompt 是否有场景化标题？
- [ ] 是否按渐进式复杂度排列？
- [ ] Prompt 是否明确目标、指定范围、提供上下文？

### 硬编码用法检查

- [ ] 是否明确区分"推荐"vs"备选"？
- [ ] 是否解释了推荐理由？
- [ ] 硬编码用法是否放在最后？
- [ ] 脚本步骤是否用编号列表组织？

### 语言风格检查

- [ ] 是否使用第二人称？
- [ ] Emoji 使用是否适度？
- [ ] 技术术语是否用代码高亮？
- [ ] 是否避免说教式文字？

### 交叉引用检查

- [ ] `` `SKILL.md` `` 引用是否准确？
- [ ] `` `config.yaml` `` 引用是否准确？
- [ ] `` `references/` `` 引用是否准确？
- [ ] 文件路径是否用正斜杠？

### 链接检查

- [ ] 所有链接是否有效？
- [ ] 相对路径是否正确？
- [ ] 锚点链接是否正确？

## 典型反模式

```

## 快速开始

```bash
python3 skills/my-skill/scripts/run.py

```
...

```
用 my-skill 做 XXX

备选用法（脚本）：
```bash
python3 skills/my-skill/scripts/run.py

```markdown

Prompt 1

```
Prompt 2
```

✅ 好：
### 示例 1：从文件提取（最简单）

```
Prompt 1
```

### 示例 2：从图片提取

```
Prompt 2
```
```

```markdown
❌ 差：
Q: 如何调整 YAML frontmatter 的 description 字段以优化语义匹配？

A: 需要考虑触发场景的覆盖范围...
