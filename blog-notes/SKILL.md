# Blog Notes — Markdown 学习笔记生成规范

## 核心目标

把用户提供的学习内容、文章摘要、课程记录、代码理解、读书收获或零散知识点，整理成结构清楚、便于以后检索和复习的 Markdown 笔记。

默认输出是一个 `.md` 文件。文件第一行必须是 YAML Front Matter 的开始标记 `---`。

## 输出结构

生成的 Markdown 文件必须使用以下顺序：

```markdown
---
category: [<分类>]
tag: [<标签>]
status: publish
---

# 前言

<说明这篇笔记记录什么、为什么值得记录、适合以后在什么场景查阅>

# 主体

<按知识点分层整理正文>

# 总结

<提炼关键结论、实践建议和后续可继续学习的问题>
```

## 参数规则

| 参数 | 生成方式 | 说明 |
|------|----------|------|
| `category` | AI 根据文章内容推断 | 文章分类，最多 2 个词语 |
| `tag` | AI 根据文章内容提取 | 内容标签，最多 2 个词语 |
| `status` | `publish` | 发布状态 |
| `filename` | AI 自动根据内容生成，或使用用户自定义文件名 | 保存的 Markdown 文件名 |

用户可以显式指定：

```text
生成 Blog-Notes 笔记：内容是...
--category AI
--tag prompt,codex
--filename codex-skill-notes.md
```

## 写作原则

1. **便于复查**: 不写成流水账，优先整理成“概念、原因、步骤、例子、注意点”。
2. **保留重点**: 保留用户给出的关键术语、命令、代码片段和结论。
3. **补齐语境**: 前言要说明这篇笔记解决什么问题，未来为什么会查它。
4. **主体分层**: 主体至少包含 2 个二级或三级小节；内容少时也要有清晰列表。
5. **总结可执行**: 总结不只复述正文，要给出 takeaway、实践建议或后续问题。
6. **不编造来源**: 没有来源时不虚构链接、作者、日期或出处。
7. **文件安全**: 文件名去除 Windows 不支持的字符：`<>:"/\|?*`。

## 主体组织方式

根据内容类型选择合适结构：

| 内容类型 | 主体建议结构 |
| --- | --- |
| 概念学习 | 定义 -> 背景 -> 核心机制 -> 示例 -> 易错点 |
| 工具使用 | 使用场景 -> 安装/前置条件 -> 操作步骤 -> 常见问题 |
| 代码理解 | 问题 -> 核心 API -> 关键流程 -> 代码片段 -> 注意事项 |
| 读书/课程 | 核心观点 -> 重要摘记 -> 我的理解 -> 可实践动作 |
| 问题复盘 | 问题现象 -> 原因分析 -> 解决方案 -> 预防清单 |

## 文件生成流程

1. 从用户输入中提取主题和主要知识点。
2. 生成或确认 `category`、`tag`、`status`、`filename`。
3. 根据内容生成文件名；正文不写文章名称或一级标题。
4. 按“前言-主体-总结”写 Markdown；正文不写文章名称，所有章节从一级标题开始。
5. 保存为 `.md` 文件；用户未指定路径时保存到当前工作目录。
6. 写入前检查目标文件是否已存在；已存在时停止并提示用户。

## 文件名规则

- 用户指定 `--filename` 时优先使用。
- 未指定时，根据内容生成简短文件名。
- 中文文件名可保留中文。
- 文件名不超过 50 个字符。
- 自动补 `.md` 扩展名。

## 输出反馈

完成后向用户说明：

```text
笔记已生成
文件: <文件名>.md
路径: <保存路径>
分类: <category>
标签: <tag>
```

## 禁止事项

- 不把 YAML Front Matter 放到文件中部或末尾。
- 不省略 `status: publish`。
- 不把主体写成单个长段落。
- 不使用虚假的来源链接。
- 不覆盖已有文件，除非用户明确要求覆盖。
